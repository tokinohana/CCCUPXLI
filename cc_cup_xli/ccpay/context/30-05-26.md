# Project Context
Last updated: 2026-05-30
Version: 1.0

## Project
Name: CCCUPXLI - CCPay Module
What it does: Internal payment processing, digital wallets, fund distribution, and expiration tracking.
Stage: MVP / In Development
Repo: local/cc_cup_xli

## Stack
Runtime: Python 3
Framework: Django 6.0
API Framework: Django REST Framework (DRF)
Task Queue: Celery (with Redis broker/backend)

## Structure
ccpay/
  models.py       <- CCPay database models
  views.py        <- Payment endpoints and views
  urls.py         <- CCPay specific routing
  services.py     <- Business logic for transactions
  tasks.py        <- Celery tasks for fund distribution & expiration
  serializers.py  <- DRF serializers for API input/output

## Database
DB: PostgreSQL (via dj-database-url)
ORM: Django ORM
Schema location: ccpay/models.py
Migrations: ccpay/migrations/
Run migrations: python manage.py migrate

## Auth
Provider: SimpleJWT
Session access: request.user via JWT Auth
Protected routes: Managed via DRF permissions

## Env Vars
DATABASE_URL
SECRET_KEY
DEBUG
CELERY_BROKER_URL
CELERY_RESULT_BACKEND

## Commands
Install: pip install -r requirements.txt
Dev: python manage.py runserver
Celery Worker: celery -A cc_cup_XLI worker -l info
Celery Beat: celery -A cc_cup_XLI beat -l info

## Conventions
- Business logic MUST be placed in `services.py`, keeping views thin.
- Scheduled tasks run via Celery Beat (e.g., daily fund distribution and expiration).
- Use DRF serializers for validation.
- All transactional database updates should use `transaction.atomic()`.

## Services
Database: PostgreSQL
Task Queue: Redis / Celery

## Gotchas
- Celery Beat schedules are predefined in `cc_cup_XLI/settings.py` for 14:00 and 17:00 Jakarta time.
- Uses JWTAuthentication exclusively (unlike regis module).

## Don't
- Don't put complex business logic in views.py; use services.py.
- Don't perform long-running processes synchronously; offload to tasks.py.
- Don't bypass the Django ORM.
